GIT - подсказки, документашки

Основные команды:

  • > git clone git@github.com:....git ./

    Клонирование репозитория с github.
    «./» означает, что создать репозиторий нужно в текущей директории.

  • > git init

    Создание (инициализация) нового репозитория в текущей директории

  • > git pull
    > git pull origin newBranch

    Вливание изменений из удаленного репозитория в локальный.
    Влить изменения из определенной удаленной ветки (в примере - из ветки newBranch)

  • > git branch
    > git branch newBranch
    > git branch -d newBranch

    "git branch" - просмотр информации о ветках, список веток влокальном репозитории (текущая ветка будет помечена звездочкой).
    Создание новой ветки с указанным именем (в примере - создание ветки newBranch).
    Удаление указанной ветки -d.

  • > git checkout newBranch
    > git checkout -b newBranch
    > git checkout --track origin/branchAnn

    Переключение на указанную ветку (в примере - на ветку newBranch).
    Создание новой ветки и переключение к ней (git checkout -b).
    Выписать (получить) уже существующую ветку с github (из удаленного репозитория), в примере - получение ветки branchAnn.

  • > git remote update

    Выполнить обновление. Если команда git checkout --track origin/.. не сработала, нужно попробовать выполнить обновление.

  • > git add .
    > git add styles/
    > git add index.html footer.php header.php ...

    Фиксирование изменений файлов, которые будут добавлены в коммит.
    Можно добавить все файлы сразу ".", отдельные каталоги со всем содержимом (в примере - styles/) или отдельные файлы (index.html, footer.php ... - через пробел)

  • > git rm index.html footer.php header.php ...

    Удаление файлов

  • > git commit -m "новый коммит"

    Создание нового коммита (сохранение изменений указанных в git add)

  • > git push
    > git push origin newBranch

    Вливание локальных изменений в удаленный репозиторий (если в master - не нужно писать "origin master").
    Если находимся не в master - указываем удаленную ветку (в примере - origin newBranch)

  • > git merge newBranch

    Слияние указанной ветки (в примере - ветки newBranch) в текущую (в которой, сейчас находимся)

  • > git status

    Информаци о текущем состоянии файлов репозиторий. Список измененных и не добавленных файлов.

  • > git log

    История коммитов репозитория с их комментариями и авторами

  • > git giff

    Подробный просмотр незафиксированных изменений в файлах

  • > git cherry-pick 300b9d9f2

    Копирование одного коммита (в примере - 300b9d9f2) из другой ветки в текущую (будет создан новый коммит в конце цепочки).

  • > git rebase

    пример:

    > git checkout newBranch
    > git rebase master

    Применение цепочки новых коммитов из указанной ветки в активную / текущую (в примере из master – в newBranch)

Краткий ход действий:

  • 1. Клонируем репозиторий "clone"
    2.1 Получаем удаленные ветку/ветки "checkout --track origin/" (если это нужно и они уже существуют) и подтягиваем в неё изменения из удаленного репозитория "pull origin"
    2.2 ИЛИ создаем новую локальную ветку "checkout -b" (если такая ветка существует, то вливаем в неё изменения из удаленного репозитория) "pull origin"
    3.1 ПСмотрим список изменений и зафиксированных / незафиксированных файлов "status"
    3.2 Поработав в ветке - фиксируем изменения "add"
    4. Коммитим изменения "commit -m"
    5. Вливаем в нашу ветку изменения из master И из ветки dev (ветка разработчиков, если такая есть)"pull origin"
    6. Отправляем изменения в удаленный репозиторий "push origin"
    7.1 Переключаемся в ветку master (для выкладки на боевой сервер) "checkout"
    7.2 ИЛИ переключаемся в ветку dev (общая ветка разработчиков и тестировщиков, именно в нее добавляются все новые разработки перед очередным релизом) "checkout"
    8. Подтягиваем изменения из удаленного репозитория "pull origin"
    9. Вливаем измения (сливаем ветки) из удаленной ветки (в которой работали ранее) в master ИЛИ в dev "merge"
    10. Отправляем изменения на удаленный репозиторий - уже из master (для обнавлений на боевом сервере) ИЛИ из Dev "push"
